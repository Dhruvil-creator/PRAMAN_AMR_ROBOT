Autonomous Mission Behavior Contract (Hardware-Safe + Synchronized)
-------------------------------------------------------------------

Core rule:
- The robot follows path **only when safe**. Safety always overrides planning and dashboard state.

System priority (must be enforced in loop order):
1. Emergency stop
2. Sensor safety validation (especially ultrasonic invalid/edge cases)
3. Gas danger
4. Hazard handling (metal)
5. Obstacle handling + replan
6. Path following

Mandatory safety gates:
- Hard stop at motor layer for unsafe conditions (no waiting for replan/confirmation).
- Unsafe means any of: near obstacle, invalid ultrasonic pattern, sudden distance anomaly, sensor inconsistency.
- Emergency cutoff rule: if any ultrasonic channel is under emergency distance, cut motor output immediately.
- Speed safety rule: speed must scale with distance; no high speed allowed near obstacle.

Ultrasonic edge-case policy (critical):
- Treat invalid as unsafe: 0, over-range, missing, abrupt jump, close->MAX blind-zone spike.
- Never trust one reading; require consistency window for "clear" before resuming.
- For close obstacle handling: stop -> validate -> mark grid obstacle -> replan -> resume.

Dashboard synchronization contract:
- Dashboard is visualization only; robot is decision authority.
- Execution is step-by-step with feedback loop:
  1. Plan accepted by robot
  2. Robot executes by real position + live sensors
  3. Robot publishes position/heading/state
  4. Dashboard renders robot-reported state
- If UI path and robot state diverge, pause and re-localize before continue.

Scenario-driven debug checklist (must pass in field):
1. Mode switch: manual loop must not interfere in autonomous.
2. Plan path: generated path visible, but execution guarded by safety.
3. Initial alignment: robot rotates to target heading before translation.
4. Normal movement: PID correction + position/heading updates every cycle.
5. Obstacle normal (<25cm): immediate stop, obstacle mark, replan, controlled resume.
6. Obstacle ultra-close (<3-5cm blind-zone risk): invalid/MAX pattern still triggers stop.
7. Metal hazard: stable detect -> stop -> servo 75->120->75 -> mark -> resume.
8. Gas danger: persistent danger triggers stop + alert + location marking.
9. Multi-event conflict: resolve by priority order above.
10. Emergency stop route: direct motor cut with no logic delay.
11. Dashboard mismatch: robot feedback is source of truth; UI must follow robot.

Required runtime evidence in logs/status:
- Safety events (`hard_stop`, `emergency_stop`, reason, distance).
- Ultrasonic raw + validated values and anomaly reasons.
- State transitions (`executing`, `paused`, `fail_safe`, `complete`).
- Obstacle/hazard mark events and replan results.
- Robot-reported position + heading stream consumed by dashboard.
